# Bugfix · 点击「文案管理」菜单首次显示空白，刷新后恢复

## 现象

从其他视图（工作台 / 需求管理）点左侧菜单「文案管理」，进入后**列表是空的**；F5 刷新页面后才有数据。

## 根因

**CSS 布局塌陷**，不是数据没渲染。

```
.list-main { display: flex; flex-direction: column; }   ← 父级是 flex 布局
└── .demand-mgr-view                                    
        既有 CSS 的 display:flex
        又被 JS 强行设为 inline style="display:block"   ❌ 优先级 inline > CSS
    └── .list-table-wrap { flex:1; overflow:auto; }
        ← 依赖父级是 flex column 才能拿到 flex:1 的高度
        ← 父级变 block 后，flex:1 失效，自身没显式高度
        ← overflow:auto + 高度为 0 → 表格被裁掉，看起来"空"
```

实际上 `cf-tbody` 里 13 个 `<tr>` 是渲染好的，浏览器自动化测试用 `document.querySelectorAll('#cf-tbody tr').length` 也确实拿到 13 行。**是渲染层 bug，不是数据 bug。**

## 修复

`scripts` 中 `showDemandMgrView()` 函数：

```javascript
// ❌ 之前
dm.style.display = 'block';

// ✅ 修复后
dm.style.display = 'flex';   // 匹配 CSS 中 .demand-mgr-view 的 display:flex
```

并加了一层防御性兜底：如果 `applyCopyFilters()` 因任何原因没正确填表，强制 fallback 到 `COPY_LIST_DATA.slice()` 后重新 render。

## 经验教训

📌 **教训 1**：用 JS 设 `style.display` 时要匹配 CSS 中该元素声明的值。如果 CSS 是 `display:flex`，JS 强行设 `block` 会把 flex 链路全部打断。

📌 **教训 2**：「数据没显示」≠「数据没渲染」。布局塌陷会让有数据的元素视觉上看不到。诊断时应该先用 `document.querySelectorAll('selector').length` 确认 DOM 层是否真的为空。

📌 **教训 3**：使用浏览器自动化测试（browser-use subagent）做诊断，比靠想象更快定位根因。
